4. Multi-currency problem: USD and ZWG
conflict in reporting
This is a serious issue and needs a more intelligent end-to-end solution.
4.1 Current problem
When documents and reports are mixing:
USD
ZWG
without a disciplined reporting/base-currency approach, reporting
becomes confusing or misleading.
Typical symptoms include:
totals that do not make intuitive sense
reports mixing transaction currency and reporting currency badly
management reports that are hard to trust
users not knowing whether a figure is:
transaction currency
converted local/base value
reporting currency value
This is especially dangerous for:
Trial Balance
Financial Position
Financial Performance
ageing reports
cashflow-style reporting
4.2 What the system should do
The system needs a proper multi-currency accounting approach from
transaction capture to reporting.
At a minimum, every transaction should support:
transaction currency
FX rate
base/local currency amount
reporting logic that is explicit and consistent
4.3 What should be true end-to-end
Rule 1: Every transaction should carry both transaction currency and
base currency values
Example:
invoice raised in USD
organisation base/reporting currency may be ZWG or USD
store:
amount in transaction currency
FX rate used
amount in base/local currency
This ensures reporting can remain consistent even when documents are
raised in multiple currencies.
Rule 2: Reporting must clearly state the currency basis
Each report should clearly support one of the following:
transaction currency view
base currency view
reporting currency view
And the report header should clearly say which basis is being shown.
Examples:
Trial Balance — Base Currency (ZWG)
Student Statement — Transaction Currency (USD)
Financial Performance — Reporting Currency (USD)
Rule 3: Management reports should not mix currencies blindly
For high-level reports such as:
Trial Balance
Financial Position
Financial Performance
cashflow
it is safest to report in one chosen reporting/base currency.
If dual-currency reporting is required, it should be an explicit feature, not
an accidental mix.
Rule 4: Operational documents may remain in transaction currency
This is normal and acceptable:
USD invoice remains USD
ZWG receipt remains ZWG
AP bill remains in supplier transaction currency
But ledger/reporting must convert consistently using stored FX rates.
Rule 5: FX rate handling must be disciplined
The system should store:
document currency
base currency
FX rate at posting time
converted local/base amount
And eventually may need:
realized exchange differences
revaluation logic
5. What likely needs to be done in IPSAS for
multi-currency
5.1 Confirm the accounting model
The system needs a clear answer to:
what is the organisation base currency?
what is the main reporting currency?
when documents are raised in another currency, when/how are they
translated?
Without this clarity, reports will remain confusing.
5.2 Standardize report behavior
For detailed reports (GL, statements, ageing)
Useful approach:
show transaction currency where relevant
show converted/base value where relevant
make totals use a clearly defined basis
For management reports
Useful approach:
default to one reporting/base currency
clearly label report currency in header
5.3 Fix mixed-currency report logic
This likely needs review in:
General Ledger totals
Trial Balance totals
Financial Position
Financial Performance
AR/AP ageing rollups
5.4 Add reporting clarity
Useful additions may include:
base currency amount columns
transaction currency amount columns
report currency selector
displayed FX rate/date basis
6. Practical recommendation for USD / ZWG
Recommended school-safe approach
For now, IPSAS should likely do the following:
let operational documents remain in their true transaction currency
force core finance reports into one chosen reporting/base currency
show transaction currency details as supporting detail, not mixed totals
This reduces confusion immediately and gives the finance team a more
reliable reporting basis.
