Section 3

Detailed Gap Analysis

Each gap below is documented with its root cause, affected files, risk level, and recommended fix. Items are ordered by priority.

1
Trial Balance Includes Unposted Vouchers
Critical Data Integrity 5-Minute Fix

The getTrialBalance() method queries all GL entries without filtering for voucher.status = "POSTED". This means draft, pending, and rejected vouchers inflate the Trial Balance figures, leading to financial misstatement.

Affected File src/lib/services/report.service.ts — lines 14–31
Root Cause The glHeader where clause only filters by organisationId and entryDate. No voucher: { status: "POSTED" } condition present.
Risk Trial Balance will not match Financial Statements (which use the same GLEntry table correctly in getFinancialStatement). Auditors will immediately notice the discrepancy.
Fix Add voucher: { status: "POSTED" } to the glHeader filter in getTrialBalance(). One line change.
Note: This is also missing from getTrialBalance() but correctly present in getFinancialStatement() — confirming this is an oversight, not an architectural decision.
2
Financial Statements Silently Return Zero if Statement Lines Not Seeded
High Reliability Setup Dependency

getFinancialStatement() relies entirely on the StatementLine and AccountStatementMap tables being correctly seeded. If a new organisation is created without running the COA seed, all three financial statements (Financial Position, Financial Performance, Cash Flow) return empty rows with zero totals — no error, no warning.

Affected Files report.service.ts:326–515, all three financial statement pages
Root Cause Architecture is correct — tree-based statement lines are powerful. But there is no validation or warning when the seed is absent.
Risk A user in a demo or new environment sees all-zero reports and assumes the system is broken or that no transactions exist.
Fix Add a _meta.mappedAccountCount field to the response. If 0, the API returns HTTP 200 with a warning: "No statement lines configured" message. The UI surfaces this with a setup prompt.
3
Stock Requisition Issue Does Not Create Inventory Movement
High Data Integrity Inventory

When a stock requisition is fulfilled (status → FULLY_ISSUED), the system updates qtyIssued on the requisition lines but does not call InventoryMovementService to decrement the item's on-hand quantity. The stock balance never decreases, making the inventory count permanently incorrect.

Affected File src/app/api/inventory/requisitions/[id]/route.ts — issue action handler
Root Cause The issue workflow was implemented at the requisition level without wiring up the downstream inventory ledger write. Two separate concerns were not connected.
Risk Inventory reports, stock take variances, and reorder calculations are all based on stale quantity data. The stock take will always show large positive variances because the system quantity is never decremented.
Fix After updating qtyIssued, call prisma.inventoryMovement.create({ type: "ISSUE", quantity: -qty, ... }) for each requisition line. Update the item's quantity field accordingly.
4
Inventory Variance Report — Page and API Missing
Medium Feature Gap 2–3 Hours

Stock takes record variances (physical qty vs system qty) but there is no dedicated report to view these variances across sessions, items, or time periods. The variance data sits in stock_take_lines but is inaccessible from the UI without opening each session individually.

Missing Files src/app/dashboard/inventory/variance-report/page.tsx, src/app/api/inventory/variance-report/route.ts
Required Data Join StockTakeLine → InventoryItem → StockTakeSession; filter where variance ≠ 0; calculate cost impact using item's average cost.
PDF Requirement The attached spec explicitly lists "Variance Report" and "Value of Variance" as required outputs from the stock governance module.
5
AR/AP Ageing: Multi-Currency FX Not Converted
Medium Multi-Currency 3–4 Hours

getARAgeing() and getAPAgeing() pass currencyCode: options.reportingCurrency directly into the Prisma where clause. This excludes invoices in other currencies from the totals entirely. A request for ZWG totals will silently omit all USD invoices.

Affected File report.service.ts:530–564
Correct Behaviour All invoices should appear regardless of currency. If the reporting currency differs from the invoice currency, the balance should be converted using the voucher's fxRate.
Fix Remove the currencyCode filter from the where clause. Include the invoice's currencyCode and voucher.fxRate in the query. In calculateAgeing(), multiply balance by fxRate when currencies differ.
6
Consolidated General Ledger (All Accounts) Missing
Medium Feature Gap 2–3 Hours

The current General Ledger report is account-specific: users must select an account before viewing entries. The PDF specification requires a consolidated view showing all accounts' movements in a date range, filterable by source module, account type, fund, and cost centre — similar to a full journal listing.

Current Behaviour GET /api/reports/general-ledger?accountId=xxx — requires a specific account ID
Required Behaviour GET /api/reports/general-ledger (no accountId) — returns all entries across all accounts, paginated, with filter params
Fix Make accountId optional in getGeneralLedger(). When absent, omit the accountId filter and group/sort by account + date. Add sourceModule and accountType filter params.
7
Stock Take Not Linked in Sidebar Navigation
Low Navigation 5-Minute Fix

The Stock Take page exists at /dashboard/inventory/stock-take but there is no sidebar link pointing to it. Users can only reach it by typing the URL directly or navigating from the Inventory overview page — if that link exists.

Affected File src/components/layout/app-sidebar.tsx — Inventory sub-section
Fix Add { title: "Stock Take", href: "/dashboard/inventory/stock-take", icon: ClipboardCheck } to the Inventory links array.
8
Demo Data Too Thin for Meaningful Report Previews
Low Demo Quality 3–4 Hours

The current seed file produces minimal data: a few invoices, one or two suppliers, and no historical GL entries. All charts appear flat or empty, which weakens the demo story. The PDF specification notes the need for realistic school-context data.

Affected File prisma/seed.ts
What's Needed 24 months of monthly journal entries (salary, utilities, school fees), 20+ inventory movements, 5+ suppliers with bill/payment history, 50+ student fee invoices across 3 terms

Section 5

Technical Architecture Notes

Key architectural patterns and invariants that must be respected when implementing the fixes above.

GLEntry Field Naming Convention

This is the most common source of bugs in the codebase. All GL entries use a four-field currency model:

FieldMeaningWhen to Use
debitLc / creditLc Local (base) currency amount — e.g. ZWG Default for all domestic reports. Use for Trial Balance, Budget vs Actuals, Departmental Expenditure
debitFc / creditFc Foreign currency amount — e.g. USD Use when reportingCurrency === "USD" or for USD bank account GL reports
fxRate Exchange rate at posting time (LC per 1 FC) Use when converting between currencies in ageing reports
Never use debit or credit — these fields do not exist on the GLEntry model. Using them will cause a TypeScript error or silent undefined values in JavaScript contexts.

GLEntry Filtering Pattern

GLEntry does not have a direct organisationId field. All organisation scoping must go through the glHeader relation:

// ✓ Correct pattern const entries = await prisma.gLEntry.findMany({ where: { glHeader: { organisationId, entryDate: { gte: startDate, lte: endDate }, voucher: { status: "POSTED" } // Always include this! } } }); // ✗ Wrong — organisationId does not exist directly on GLEntry const entries = await prisma.gLEntry.findMany({ where: { organisationId } // TypeScript error or runtime failure });

Auth Patterns

PatternUsageReturns
withAuth(req, handler, roles[]) Older routes; role-gated middleware Injects authReq.user.{ userId, organisationId, role }
verifyAuth(req) Newer report routes Returns { userId, organisationId, role } or null

StockTakeStatus Lifecycle

DRAFT → COUNTED → APPROVED → POSTED
  • DRAFT: Session created, lines added, physical counts entered. Editable.
  • COUNTED: Counts saved, variances calculated. Locked from editing.
  • APPROVED: Approved by ADMIN/BURSAR/HEADMASTER. Ready to post.
  • POSTED: Variance adjustments written to InventoryMovement. Immutable.

Multi-Currency Rule Summary

  • GL reports use useFc = reportingCurrency === "USD" flag to pick Lc vs Fc fields
  • Financial statements correctly pass reportingCurrency through the service chain
  • Budget vs Actuals always uses debitLc (local currency base)
  • AR/AP Ageing excludes foreign-currency invoices from totals when currency filter is applied
  • Consolidated GL does not exist yet — multi-currency support to be added alongside it
  • No FX rate conversion utility function exists — each service handles it ad-hoc

Prisma Migration Notes

This project uses prisma db push (schema push) rather than migrations. This means there is no migration history file. For destructive operations (dropping columns, renaming), prisma db push --force-reset is required with the PRISMA_USER_CONSENT_FOR_DANGEROUS_AI_ACTION=true environment variable set. Always re-seed after a force reset.
